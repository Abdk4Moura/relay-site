We drove a BIOS screen from a phone.

Remote desktop tools give up at the login screen. Every one of them needs something running inside a booted, logged-in operating system. The moment the OS is not there, the moment you are sitting at a BIOS, a boot menu, or a machine that just hung on a black screen, the tool you trust to reach a computer reaches nothing.

We wanted to go lower than that. So we built Relay, and last week we used a phone to step through a BIOS, change a boot order, and save and exit. Nothing was installed on the target. The target did not have an operating system to install into.

What Relay is

Relay turns a phone into a keyboard, mouse, and HID controller for any machine. There is nothing to install on the machine you are controlling. You pick it up, you point it at a target, and you type.

The piece that does the work is a small cross-platform receiver, and it injects real input at the lowest layer each platform offers:

  • Linux through the kernel's uinput device.
  • Windows through SendInput.
  • macOS through CGEvent.

These are not scripted screenshots of input. They are the same events the kernel and the window server see when you press a physical key.

How we know it works

I do not trust an input library because the docs say it works. I trust it when I can read the event back off the other side. So that is what we did, on each platform, by different means.

On Linux, the path is verified end to end in CI. We inject events through uinput, then read them straight back off the kernel input device and assert that the bytes match what we sent. Every commit proves the keyboard works before it ships.

On Windows, we drove a live machine over an RDP session through a Tailscale tunnel and watched the keystrokes land in real time. A real desktop, a real network, real typing.

On macOS, we did not have a desktop display in front of us, so we proved it the honest way: we read the keystrokes off the wire as they were sent. The events were correct on the line.

Three platforms, three independent proofs. No hand-waving.

The idea that turned out to matter

Here is the part we did not expect to care about as much as we do now.

Software remote control fails wherever there is no software to run. But a person sitting at the machine does not have that problem. They use their hands and their eyes. So give the remote a body. HID for the hands. Video capture for the eyes. A small bridge taps the target's video through a capture card and presents itself to the target as a plain USB keyboard.

The target sees a monitor cable and a keyboard. That is all it can see. And because that is all it needs to see, this reaches places an in-band tool can never go: the login screen, the BIOS, a smart TV, a server whose OS is dead. This is remote control for the machines remote control can't reach.

That is the out-of-band capability, and it is the direction the BIOS demo points at. We wrote up the full method and what it means for hardware-level control in a separate piece, Out-of-Band Computer Use. If you want the deep technical version of the capture-and-emulate path, start there.

What ships today, plainly

I will be exact, because launch posts usually are not.

What you can use right now, on your own network, is keyboard, mouse, and HID control of a target, two-way file transfer between phone and machine, and a web panel to drive it all. That is real and it is in your hands today.

What is not shipped yet is live video streaming of the target's screen back to the phone. That is on the roadmap. The out-of-band BIOS rescue, the full eyes-and-hands experience over a capture card, is the proven direction, not a button you can press in the app this week. I would rather tell you that now than have you find out later.

A body for agents

The same thing that lets a phone drive a locked-out machine lets something else drive it too. Relay exposes its input and control as MCP tools, so a computer-use agent can act on a machine through the exact same path a person does.

That means an agent can operate a machine it could never be installed inside. A BIOS. A boot menu. An appliance with no shell. The agent asks for the screen and sends input. It does not care whether the target is a booted desktop or a box stuck at firmware. Giving agents a body that reaches real hardware is, to us, the most interesting thing here.

It is open source

Relay is GPLv3. The receiver is the thing that types on your machine, so you should be able to read exactly what it does. The two repos are relay-desktop and relay-hid.

You can download the receiver and read the docs at relay.autumated.com. On your own network it is free. Pick it up, point it at a machine you are locked out of, and type.
