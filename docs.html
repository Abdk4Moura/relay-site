Getting started

Relay has two pieces: the app on your phone, and a receiver on the machine you want to control. On your own network, both are free.

  1. Install the Relay app on your phone.
  2. Install the open-source receiver on the machine you want to control.
  3. Open the app. The machine appears in your device list. Tap to connect.
# macOS / Linux, install the receiver
curl -fsSL relay.autumated.com/get | sh
relay-receiver start # now visible to your phone on the LAN

No account needed for local control. You only sign in when you want to reach machines from outside your network, which is a Pro feature.

The two modes

Relay is honest about a trade-off. Choose the mode that matches the target.

App mode

The receiver runs inside a booted, logged-in OS. This gives you the richest experience: live screen, trackpad, keyboard, and two-way file transfer. It is the right choice for your own desktop, a media PC, or a server you can already log into. The receiver is open source so you can read exactly what it does.

Out-of-band mode

Nothing runs on the target. A small bridge taps the target's video through a capture card and presents itself to the target as a USB keyboard. Because the target only sees a keyboard and a monitor cable, this mode reaches places no agent can live: the BIOS, the boot menu, a login screen, a smart TV, or a machine whose OS has crashed.

Rule of thumb: if you can already log in, use App mode. If you are locked out, powered down to firmware, or on a device that will never run a receiver, use Out-of-band mode.

Security model

Local control stays on your network and is encrypted end to end. The receiver is open source and auditable, which matters because it is, by design, something that types on your machine. Reaching machines from outside your network routes through the hosted relay, which handles NAT traversal and never holds your keys.

Build the out-of-band bridge

This is the full do-it-yourself build. By the end you will have a Raspberry Pi that appears to a target machine as an ordinary USB keyboard and mouse, captures that machine's HDMI output, and lets your phone drive it over WiFi. Everything below is plain Linux: a USB gadget through configfs and libcomposite, and video through V4L2 and ffmpeg. You can build it from a stock Raspberry Pi OS image without the prebuilt firmware.

1 · What it is and why

The bridge is the body of out-of-band control. In App mode a receiver runs inside a booted, logged-in OS. But there are places no software can live: a BIOS or UEFI setup screen, a boot menu, a full-disk-encryption password prompt, a frozen login screen, a smart TV, or a box whose OS has died. Nothing on the target can run your code, so the control has to come from outside the target entirely.

The Pi is that outside. Your phone talks to the Pi over WiFi. The Pi does two physical things to the target: it pretends to be a USB keyboard and mouse plugged into a USB port, and it watches the target's screen through an HDMI capture dongle. To the target, that is just a keyboard and a monitor cable. Firmware accepts a USB keyboard natively, before any operating system exists, which is exactly why this reaches the BIOS.

The Pi is the hands and eyes; your phone is the brain. The target never runs Relay and never knows the Pi is anything but human-attached peripherals.

2 · Parts list

Buy these before you start. The two parts that matter most are the Pi (it must support USB gadget mode) and the capture dongle (it must be a UVC device).

  • A Raspberry Pi with USB OTG / gadget support. A Pi Zero 2 W is ideal: its micro-USB data port is wired straight to the SoC USB and does gadget mode natively, and it has WiFi. A Pi 4 also works, but you must use its single USB-C port as the gadget port (the four USB-A ports are host-only). The Pi 3 and most Pi-clone boards cannot do gadget mode on a useful port; check the board before buying.
  • A microSD card, 16 GB or larger, for Raspberry Pi OS.
  • An HDMI-to-USB capture dongle, UVC class. The common "MS2109" style USB 2.0 sticks are UVC and cost a few dollars. UVC matters: it means Linux exposes it as a standard /dev/video0 with no driver to install. Avoid anything that ships a Windows-only capture app, since those are rarely UVC.
  • Cabling. An HDMI cable from the target's HDMI out into the capture dongle, and a USB cable from the Pi's gadget port into one of the target's USB ports. On the Pi Zero 2 W that gadget cable is micro-USB; on the Pi 4 it is USB-C. The gadget cable carries both the HID data and bus power, so the target also powers the Pi, which is convenient at a BIOS where the Pi has no other supply.

Test the dongle before you trust it. Cheap UVC capture sticks were built for game consoles at clean 1080p. A BIOS often outputs odd low resolutions (640x480, 720x400 text modes) and changes mode several times during POST. Some dongles drop the signal or hand you a black frame on those modes. Verify yours holds a picture through a real power-on before you rely on it for a job. See Capture that survives POST for the tested list.

3 · The HID gadget

This is the half that lets the Pi type. Linux can present a USB device to whatever it is plugged into using the gadget subsystem. You build a composite gadget with two HID functions, a keyboard and a mouse, each with its own report descriptor, then bind it to the USB device controller (UDC). After this, plugging the Pi's gadget port into the target makes the target enumerate a keyboard and a mouse.

First, enable the device-tree overlay and the modules. On a Pi Zero 2 W add the OTG overlay to the boot config and load libcomposite at boot.

# /boot/firmware/config.txt (or /boot/config.txt on older images)
dtoverlay=dwc2

# /etc/modules
dwc2
libcomposite

Now create the gadget under configfs. Run this once at boot (a systemd unit or /etc/rc.local). The gadget lives entirely in the filesystem at /sys/kernel/config/usb_gadget/; you create it by making directories and writing files.

#!/bin/bash
modprobe libcomposite
cd /sys/kernel/config/usb_gadget/
mkdir -p relay && cd relay

# identify as a generic composite HID device
echo 0x1d6b > idVendor # Linux Foundation
echo 0x0104 > idProduct # Multifunction composite
echo 0x0100 > bcdDevice
echo 0x0200 > bcdUSB # USB 2.0
mkdir -p strings/0x409
echo "relay-bridge" > strings/0x409/serialnumber
echo "Relay" > strings/0x409/manufacturer
echo "Relay HID Bridge" > strings/0x409/product

mkdir -p configs/c.1/strings/0x409
echo "HID" > configs/c.1/strings/0x409/configuration
echo 250 > configs/c.1/MaxPower

Add the keyboard function. The report descriptor below is the standard boot keyboard: one byte of modifiers, one reserved byte, and six key slots. This is the descriptor a BIOS expects, so it works before any OS loads.

# keyboard HID function
mkdir -p functions/hid.kbd
echo 1 > functions/hid.kbd/protocol # 1 = keyboard
echo 1 > functions/hid.kbd/subclass # 1 = boot interface
echo 8 > functions/hid.kbd/report_length # 8-byte reports
# write the 63-byte boot-keyboard report descriptor
echo -ne \x05\x01\x09\x06\xa1\x01\x05\x07\x19\xe0\x29\xe7\x15\x00\x25\x01\x75\x01\x95\x08\x81\x02\x95\x01\x75\x08\x81\x03\x95\x05\x75\x01\x05\x08\x19\x01\x29\x05\x91\x02\x95\x01\x75\x03\x91\x03\x95\x06\x75\x08\x15\x00\x25\x65\x05\x07\x19\x00\x29\x65\x81\x00\xc0 > functions/hid.kbd/report_desc

Add the mouse function. This descriptor reports three buttons plus relative X and Y, which is all you need to move the pointer and click.

# mouse HID function
mkdir -p functions/hid.mouse
echo 2 > functions/hid.mouse/protocol # 2 = mouse
echo 1 > functions/hid.mouse/subclass
echo 4 > functions/hid.mouse/report_length # buttons + dx + dy + wheel
echo -ne \x05\x01\x09\x02\xa1\x01\x09\x01\xa1\x00\x05\x09\x19\x01\x29\x03\x15\x00\x25\x01\x95\x03\x75\x01\x81\x02\x95\x01\x75\x05\x81\x03\x05\x01\x09\x30\x09\x31\x09\x38\x15\x81\x25\x7f\x75\x08\x95\x03\x81\x06\xc0\xc0 > functions/hid.mouse/report_desc

Link both functions into the config and bind the gadget to the UDC. Binding is the moment the Pi goes "live" on the USB bus; the target enumerates it instantly.

ln -s functions/hid.kbd configs/c.1/
ln -s functions/hid.mouse configs/c.1/

# bind to the first available USB device controller
ls /sys/class/udc > UDC
# now /dev/hidg0 is the keyboard, /dev/hidg1 is the mouse

After binding you have two character devices. Writing an 8-byte report to /dev/hidg0 sends a keystroke; writing a 4-byte report to /dev/hidg1 moves or clicks the mouse. A quick sanity check, with the gadget cable in a target you can watch: this taps the a key (USB usage 0x04) and releases it.

# press 'a' (no modifiers), then release all keys
printf '\\0\\0\\x04\\0\\0\\0\\0\\0' > /dev/hidg0
printf '\\0\\0\\0\\0\\0\\0\\0\\0' > /dev/hidg0

4 · Capturing the screen

This is the half that lets the Pi see. Plug the HDMI from the target's output into the capture dongle, and the dongle into one of the Pi's USB host ports (on the Pi Zero 2 W that is the second micro-USB port through an OTG adapter; on the Pi 4 any USB-A port). Because the dongle is UVC, it shows up as a V4L2 device with no driver work.

# confirm the dongle enumerated as a V4L2 capture device
v4l2-ctl --list-devices
v4l2-ctl -d /dev/video0 --list-formats-ext # what resolutions/timings it offers

Pull frames with ffmpeg. Reading from /dev/video0 as a v4l2 input gives you the live target screen; encode it to MJPEG so each frame stands alone and the phone can show video as fast as the link allows.

# grab the target's screen as an MJPEG stream
ffmpeg -f v4l2 -framerate 30 -video_size 1280x720 \
  -i /dev/video0 -f mjpeg -q:v 5 pipe:1

# or one still frame, useful for testing the capture path
ffmpeg -f v4l2 -i /dev/video0 -frames:v 1 /tmp/target.jpg

If the still is black or garbled at a BIOS but fine in a booted OS, the dongle is choking on the firmware's video mode, not on your wiring. That is the single most common failure in this whole build. Drop to --video_size 640x480 and let ffmpeg accept whatever timing the dongle reports.

5 · Running the Relay bridge on the Pi

You now have the two raw capabilities: write to /dev/hidg0 and /dev/hidg1 to act, read /dev/video0 to see. The bridge process glues them to the network. It accepts input from your phone over WiFi or Tailscale, translates each event into the right HID report and writes it to the gadget, and serves the captured frames back the other way. This is the same shape as the App-mode receiver, which turns wire-protocol key and pointer frames into synthetic input; the only difference is that here the "input backend" is the USB HID gadget instead of uinput, and the screen comes from the capture dongle instead of a framebuffer grab.

If you are running the prebuilt firmware image, the relay-bridge service is already wired to those devices and starts on boot. If you built the gadget yourself by hand, point the bridge at the two HID devices and the video node:

# the bridge maps wire-protocol frames to HID reports and serves video
relay-bridge run \
  --keyboard /dev/hidg0 \
  --mouse    /dev/hidg1 \
  --video    /dev/video0 \
  --listen   tailscale0 # or your LAN interface

It speaks the same open wire protocol as everything else (see Wire protocol): a key frame becomes an 8-byte keyboard report, a pointer frame becomes a 4-byte mouse report, and a video.req frame turns on the MJPEG feed.

6 · Pair the phone and drive the target

With the gadget bound, the capture running, and the bridge listening, pair your phone. The bridge prints a code; enter it once in the app.

# on the Pi
relay-bridge pair # shows a code to enter in the phone app
relay-bridge status
video: capture ok · 720p30
hid:   keyboard + mouse bound to UDC
net:   listening on tailscale0

Now plug the two cables into the target: capture dongle to the target's HDMI out, Pi gadget port to a target USB port. Power on the target. Because the Pi is a boot-class USB keyboard, you can hammer the BIOS hotkey (Del, F2, F12, whatever your board uses) from the phone the instant the target powers on, navigate the firmware menus with the arrow keys, and watch every POST screen on the phone through the capture feed. No OS, no agent, no cooperation from the target required.

7 · Honest caveats

  • Capture dongles are the weak link. Cheap UVC sticks can miss or mangle the low-resolution, fast-changing video a BIOS emits. Test through a real power-on, keep a known-good dongle for firmware work, and consult the compatibility list before buying.
  • You spend one target USB port. The gadget needs a real USB port on the target. On a machine with few ports, plan for it.
  • You need physical access once. Somebody has to plug in two cables at the target: HDMI out to the dongle, USB to the gadget port. There is no way around the initial wiring.
  • After that, you can be remote. Once the Pi is wired and on your network, the operator does not have to be in the room. Put the Pi on Tailscale and you can sit at a BIOS screen from anywhere, which is the whole point of leaving a bridge installed on a machine you cannot otherwise reach.
  • It is genuinely a keyboard, for better and worse. The target trusts it completely, which is the power of the approach and also why you should only attach a bridge to machines you are authorised to control.

Capture that survives POST

Many cheap capture sticks only initialise once the source sends a clean signal, which means they miss the BIOS entirely. For out-of-band work you want a card that locks signal early and tolerates resolution changes during boot. The firmware repo keeps a community-tested compatibility table.

Fleet setup

On the Team plan, every bridge and receiver can register into a shared directory. Operators get roles, connections are logged, and you can revoke a device in one click.

Wire protocol

The protocol that carries input, video, and files between the app, the relay, and the receiver or bridge is open and documented. You can build your own client or your own receiver against it. It is transport-agnostic and versioned.

// a frame is a typed envelope
{ "v": 1, "t": "key", "code": "Enter", "mods": ["gui"] }
{ "v": 1, "t": "pointer", "dx": 12, "dy": -4, "btn": 0 }
{ "v": 1, "t": "video.req", "fps": 30 }

Agent / MCP

Relay exposes an MCP server so computer-use agents can drive a target through the same protocol: read the live screen, send keys and pointer events, wait for the screen to settle. This is a Pro feature, and it is the same path whether the target is a booted desktop or a locked-out box on the out-of-band bridge.

An agent does not care which mode it is using. It asks for the screen and sends input. Relay handles the rest.
