Getting started

Relay has two pieces: the app on your phone, and a receiver on the machine you want to control. On your own network, both are free.

  1. Install the Relay app on your phone.
  2. Install the open-source receiver on the machine you want to control.
  3. Open the app. The machine appears in your device list. Tap to connect.
# macOS / Linux, install the receiver
curl -fsSL relay.autumated.com/get | sh
relay-receiver start # now visible to your phone on the LAN

No account needed for local control. You only sign in when you want to reach machines from outside your network, which is a Pro feature.

The two modes

Relay is honest about a trade-off. Choose the mode that matches the target.

App mode

The receiver runs inside a booted, logged-in OS. This gives you the richest experience: live screen, trackpad, keyboard, and two-way file transfer. It is the right choice for your own desktop, a media PC, or a server you can already log into. The receiver is open source so you can read exactly what it does.

Out-of-band mode

Nothing runs on the target. A small bridge taps the target's video through a capture card and presents itself to the target as a USB keyboard. Because the target only sees a keyboard and a monitor cable, this mode reaches places no agent can live: the BIOS, the boot menu, a login screen, a smart TV, or a machine whose OS has crashed.

Rule of thumb: if you can already log in, use App mode. If you are locked out, powered down to firmware, or on a device that will never run a receiver, use Out-of-band mode.

Security model

Local control stays on your network and is encrypted end to end. The receiver is open source and auditable, which matters because it is, by design, something that types on your machine. Reaching machines from outside your network routes through the hosted relay, which handles NAT traversal and never holds your keys.

Build the bridge

The bridge is any Linux board with a USB gadget mode and a free USB port for a capture card. A Raspberry Pi Zero 2 W or a Pi 4 both work. Flash the firmware image, then point it at the target.

# on the Pi, after flashing the Relay bridge image
relay-bridge pair # shows a code to enter in the phone app
relay-bridge status
video: capture ok · 1080p60
hid: usb gadget attached

Plug the capture card into the target's HDMI out, and the gadget port into one of the target's USB ports. The target sees a monitor and a keyboard. Nothing else.

Capture that survives POST

Many cheap capture sticks only initialise once the source sends a clean signal, which means they miss the BIOS entirely. For out-of-band work you want a card that locks signal early and tolerates resolution changes during boot. The firmware repo keeps a community-tested compatibility table.

Fleet setup

On the Team plan, every bridge and receiver can register into a shared directory. Operators get roles, connections are logged, and you can revoke a device in one click.

Wire protocol

The protocol that carries input, video, and files between the app, the relay, and the receiver or bridge is open and documented. You can build your own client or your own receiver against it. It is transport-agnostic and versioned.

// a frame is a typed envelope
{ "v": 1, "t": "key", "code": "Enter", "mods": ["gui"] }
{ "v": 1, "t": "pointer", "dx": 12, "dy": -4, "btn": 0 }
{ "v": 1, "t": "video.req", "fps": 30 }

Agent / MCP

Relay exposes an MCP server so computer-use agents can drive a target through the same protocol: read the live screen, send keys and pointer events, wait for the screen to settle. This is a Pro feature, and it is the same path whether the target is a booted desktop or a locked-out box on the out-of-band bridge.

An agent does not care which mode it is using. It asks for the screen and sends input. Relay handles the rest.
